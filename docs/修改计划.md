# 修改计划（待审阅）

面向目标：首启登录与向导、默认行为对齐预期、开箱即用（分期）、Electron 稳定性、通知仅在向导与设置中配置、登录统一走应用内窗口。  
原则：**少写新 IPC、新状态源**；偏好与主题继续以 **`electron_prefs.json` + 现有 `get/set-startup-prefs`** 为单一事实来源；通知逻辑仍只在 **`main.js` 的 `runAlertCheck` / `restartAlertScheduler`**；首启只做 **编排与 UI**，不复制一份预警算法。

---

## 一、原则（简洁与安全冗余）

| 原则 | 做法 |
|------|------|
| 不重复造轮子 | 首启页能调用的能力全部复用 `preload` 已暴露的 `window.buptHw`；配置项与设置页共用同一套偏好键。 |
| 单一数据源 | 登录态仍以 `playwright_storage_state.json` + `has-login-session` 为准；同步间隔仍以 `config.yaml` 的 `widget_refresh_minutes` 为准。 |
| 安全冗余 | 关键路径（导出会话、写 prefs）保持现有 try/catch 与主进程校验；首启仅增加 **幂等标记**（见下文），避免重复弹向导。 |
| 兼容老用户 / 重装 | 调整「默认值」时，仅对 **从未写过 prefs** 或 **首启未完成** 的用户生效；已有 `electron_prefs.json` 的用户不被静默改写行为。**应用重装**时：用户数据目录（打包后 `app.getPath('userData')`，与可执行文件分离）**应保留** prefs、登录态、缓存等；若检测到 **已完成过首启** 或 **已有本机数据**，则 **不再显示向导**。 |

---

## 二、分期范围

### 阶段 A：首启流程 + 默认行为（不接环境捆绑）

**目标**

1. 首次安装/首次运行：先完成 **应用内登录**（沿用现有 `login-shell` + `export-login-session`），再进入 **简化配置步**（具体内容待你定稿；可先占位：主题、通知总开关与档位说明、链到完整设置）。
2. 默认偏好：`openAtLogin: true`，`autoSyncOnLaunch: true`（后者已与现状一致则不动）。**进入桌面时打开哪个窗口**不设死：增加偏好项（例如 `startupOpenMode`：`home` | `widget` | `both`），由用户在 **设置页** 选择；首启向导里提供同一选项（写同一键），默认可先设为 `widget` 或与当前「仅主页」对齐后由你定 **仅作首次安装的默认值**。
3. 向导内与设置页 **共用** 通知相关控件：读写同一 prefs，避免两套 UI 两套逻辑。

**实现要点（避免臃肿）**

- 在 `electron_prefs` 增加一项即可区分首启是否完成，例如 `onboardingDone: false`，完成向导后置 `true`。主进程 `app.whenReady` 里：**仅当**「未完成首启」且 **需要向导** 时再进入向导流程；**重装或升级**后若磁盘上仍存在 `onboardingDone: true`（或等价判定：已有 prefs + 曾登录），则 **跳过向导**，直接进入按偏好启动窗口。
- **重装保留信息**：数据与安装包分离（Electron 惯例）；安装器/卸载程序 **不应默认清空** `%APPDATA%` 下本应用 `userData`（需在 `electron-builder`/NSIS 侧约定，阶段 A 实施时核对）。用户若手动删除用户目录则视为新机，可再走首启。
- **推荐**：新增 **一个** 轻量窗口或 **主页内全屏遮罩页**（`electron/onboarding/`），仅负责步骤跳转；登录步直接 `openLoginWindow()` 或内嵌与现有登录页相同的流程，**不复制** `login-shell` 业务代码。
- 调整启动顺序时，只改 `main.js` 里 `whenReady` 末尾：根据 **`startupOpenMode`**（或最终命名）分支调用 `createHomeWindow` / `createWidgetWindow`，逻辑集中在一处；设置页与首启通过 `set-startup-prefs` 写入，**不新增专门 IPC**（除非枚举校验有必要在主进程再做一层）。

**涉及文件（预估）**

- `electron/main.js`：启动分支、`DEFAULT_STARTUP_PREFS`、可选 `get-onboarding-state` / `set-onboarding-done`（若不愿加 IPC，可由现有 `set-startup-prefs` 写入 `onboardingDone`）。
- `electron/preload.js`：若首启页需要只读状态，优先用已有 `get-startup-prefs` / `has-login-session`，尽量不新增 API。
- 新建 `electron/onboarding/*`（可选）或扩展 `home`，保持页面数最少。
- `electron/settings/*`：核对与向导字段重复时，抽 **共用文案或小函数**（仅前端），不把业务逻辑拆两份。

**已定**

- ~~启动窗口组合~~ → **用户可在设置（及首启）中选择**，见上。
- ~~重装 / 老用户是否再看向导~~ → **重装保留原有用户信息**（用户数据目录不随卸载清空为原则）；**不再显示向导**。若 prefs 中已有 `onboardingDone: true`，或检测到已有登录会话且 prefs 完整，均不再强制首启。

### 阶段 B：开箱即用（Python / Playwright 环境）

**目标**：安装或首次运行后 **自动** 完成可运行的 Python 依赖与 Chromium，无需用户手动 `pip` / `playwright install`。

**原则**

- 优先 **商用成熟路径**：Windows 下嵌入 **嵌入式 Python**、固定版本 `requirements.txt`、安装器或首启子进程执行 `pip install`；Playwright 使用 **打包的浏览器路径** 或 `playwright install chromium` 安装到 `userData` 下固定目录，通过环境变量让 `app.py` / Playwright 使用（与现有 `getPythonPath`、`BUPT_DATA_DIR` 对齐，**扩展**而非重写）。
- `getPythonPath()` / `getPythonAppPath()` 保持为 **唯一解析入口**，新增探测分支即可，避免多处拼路径。

**交付形态（择一或组合）**

- NSIS / 安装步骤中静默安装依赖；或
- 首启向导最后一步「准备环境中…」调用已有 `spawn` 封装（可抽 `runEnvSetup()` 与 `runFetch` 并列），统一日志与错误展示。

**依赖**：阶段 A 稳定后再做，避免两条线同时改启动路径。

---

### 阶段 C：内存与长期运行（非一次性「改一行」）

**目标**：降低 **长时间运行 + 多窗口** 下的内存爬升与异常退出风险。

**原则**

- **先度量后优化**：主进程可选记录窗口数、`webContents` 内存（按需），避免无证据盲改。
- **低成本手段**：不用的窗口 `destroy`；登录窗关闭后清理引用（现有 `loginWindow` 已部分处理）；限制 **重复打开** 同一类窗口（设置/登录单例模式已有则沿用）。
- **渲染进程**：列表过长时对小组件列表做 **简单上限或虚拟化**（若当前已全量 DOM，再评估）；避免在 `setInterval` 里堆积闭包与大对象。

**不做**：过早引入重型监控框架；与阶段 B 的 Python 子进程内存分列排查。

### 阶段 D：文档与入口收敛

**目标**：用户主路径 **只在 Electron 内登录**；`首次登录.bat`、`bupt-hw.cmd login` 移到 **开发者/排错** 章节，避免与产品流程混淆。

**涉及**：`README.md`、`docs/项目说明.md`、必要时 `HANDOFF.md` 一句指向。

## 三、通知与设置（与你期望对齐）

- **逻辑唯一**：仍只在 `main.js` 中 `DEFAULT_STARTUP_PREFS`、`runAlertCheck`、`restartAlertScheduler`。
- **UI 两处**：首启（简版）+ 设置（完整）→ 同一 prefs 键，设置页为 **主编辑界面**；首启可为「快捷开关 + 链到设置」。
- **启动打开谁**：与通知同理——**偏好键唯一**，设置与首启共用；用户随时在设置里改，下次启动生效。

## 四、风险与回滚

| 风险 | 缓解 |
|------|------|
| 改默认开机启动打扰用户 | 首次安装向导中明确勾选含义；老用户不覆盖已有 prefs。 |
| 首启阻塞无法进入应用 | 提供「跳过（仅限登录失败排错）」或后台完成标记策略（产品定）。 |
| 卸载清空数据导致「像新机」 | 文档说明：手动删用户目录或勾选清除数据后会重新首启；非默认行为。 |
| 阶段 B 安装失败 | 失败时回落到当前「自备 Python」提示，与今日行为一致。 |

---

## 五、建议实施顺序

1. **阶段 A**（首启 + 默认行为 + prefs 标记），联调 `npm start` 与 `npm run dist` 安装路径。  
2. **阶段 D**（文档入口收敛），与 A 同时或紧随。  
3. **阶段 B**（开箱即用），专项分支/版本。  
4. **阶段 C**（内存），持续迭代，可与 B 并行观测。
